## рҹ—„пёҸ How the App & Database Work

The core purpose of Romulus is simple: **compare ROM files on your hard drive against what a DAT file says should exist**.  
The schema is built around that single question.

### The tables and their roles

| Table | What it stores | Why it exists |
|-------|---------------|---------------|
| `dat_versions` | Each imported DAT file (name, version, system description, SHA-256 checksum). Unique by checksum so the same file can't be imported twice. | The rulebook вҖ” what a given publisher decided correct ROMs look like. |
| `games` | Normalized game entries вҖ” one row per unique game name per DAT version. | Eliminates duplication: 10 ROMs for the same game share one `games` row rather than duplicating the name 10 times. |
| `roms` | The **expected** ROM entries declared by a DAT (`expected_sha1`, `crc32`, `md5`, size). Each ROM is a child of a `games` row via `game_id` FK. | *Opinion* вҖ” what the DAT author says a correct ROM looks like. |
| `files` | Every ROM **file found on disk** (virtual path, archive_path, optional entry_name, size, hashes, Unix scan timestamp). Points into `global_roms` via `sha1`. | *Reality* вҖ” what is actually sitting in your scan folders. Archive entries are first-class: `archive_path` + `entry_name` are stored separately. |
| `global_roms` | **Content-addressable file identity** keyed by SHA-1. Multiple paths can map to the same content blob. | Deduplication вҖ” the same binary in two folders is one `global_rom`, two `files`. |
| `rom_matches` | Which `global_rom` satisfies each `rom`, and *how* (`match_type` integer enum). | The verdict per ROM вҖ” populated by the matcher, queried by the classifier. |
| `scanned_directories` | User-registered scan folders, persisted across sessions. | Remember where to look without re-adding every launch. |

### How they connect

```text
dat_versions в”Җв”Җ< games в”Җв”Җ< roms
                              в”Ӯ
                              в””в”Җв”Җ rom_matches в”Җв”Җ> global_roms <в”Җв”Җ files
                                   (match_type)   (SHA-1 PK)    (path on disk)
```

`rom_matches` is the **bridge**: for every `rom` (expected), it records which `global_rom` (actual) satisfies it, and *how* it was matched:

| `match_type` | Value | Meaning |
|---|---|---|
| `Exact` | 0 | CRC32 + MD5 + SHA1 all agree вҖ” gold standard |
| `Sha256Only` | 1 | Only SHA-256 matches (enriched DAT entry) |
| `Sha1Only` | 2 | Only SHA-1 matches |
| `Md5Only` | 3 | Only MD5 matches |
| `Crc32Only` | 4 | Only CRC32 matches |
| `SizeOnly` | 5 | Only file size matches вҖ” weakest signal |
| `NoMatch` | 6 | No match found |

### Workflow вҖ” step by step

```text

в”Ӯ  1. IMPORT DAT                                                        в”Ӯ
в”Ӯ     romulus import-dat "Nintendo - Game Boy (20240101).dat"           в”Ӯ
в”Ӯ                                                                       в”Ӯ
в”Ӯ     Parses the LogiqX XML вҶ’ inserts:                                  в”Ӯ
в”Ӯ       dat_versions (one row)                                          в”Ӯ
в”Ӯ       games        (one row per unique game name)                     в”Ӯ
в”Ӯ       roms         (one row per ROM entry, linked to its game)        в”Ӯ
в”Ӯ                                                                       в”Ӯ
в”Ӯ  2. SCAN FOLDERS                                                      в”Ӯ
в”Ӯ     romulus scan /path/to/roms/GameBoy                                в”Ӯ
в”Ӯ                                                                       в”Ӯ
в”Ӯ     Walks the directory, hashes each file (CRC32+MD5+SHA1 in one     в”Ӯ
в”Ӯ     pass) вҶ’ inserts/updates:                                          в”Ӯ
в”Ӯ       global_roms  (content identity, keyed by SHA-1)                в”Ӯ
в”Ӯ       files        (path вҶ’ global_rom link)                          в”Ӯ
в”Ӯ     Already-known files with unchanged size/mtime are skipped.       в”Ӯ
в”Ӯ     Archive files (zip/7z) are opened and each entry hashed.         в”Ӯ
в”Ӯ                                                                       в”Ӯ
в”Ӯ  3. MATCH  (romulus verify, step 1)                                   в”Ӯ
в”Ӯ     Compares every rom against every global_rom in priority order:   в”Ӯ
в”Ӯ       SHA-1 вҶ’ SHA-256 вҶ’ MD5 вҶ’ CRC32 вҶ’ size                          в”Ӯ
в”Ӯ     вҶ’ inserts rom_matches rows with the match_type verdict           в”Ӯ
в”Ӯ                                                                       в”Ӯ
в”Ӯ  4. CLASSIFY  (romulus verify, step 2)                                в”Ӯ
в”Ӯ     Reads rom_matches + files via CTE вҖ” no separate status table:    в”Ӯ

```

ROM **status is never stored** вҖ” it is computed live from `rom_matches` + `files` so it can never go stale.

### Verification flow (condensed)

```text
1. Import DAT   вҶ’ dat_versions + games + roms      (expectation)
2. Scan folders вҶ’ global_roms + files              (reality)
3. Match        вҶ’ rom_matches                      (verdict per ROM)
4. Classify     вҶ’ CTE over rom_matches + files     (Verified/Missing/Unverified/Mismatch)
5. Report       вҶ’ reads classified results         (Text / CSV / JSON)
```
